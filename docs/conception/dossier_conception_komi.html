1. Présentation du projet

Komi est une application mobile qui aide l'utilisateur à préparer ses courses, importer une liste, identifier les aliments, consulter un bilan nutritionnel et découvrir des recettes.

Le coeur du MVP est la liste de courses. L'utilisateur peut saisir des aliments, importer une image de liste via caméra ou galerie, puis laisser l'application extraire et filtrer les éléments alimentaires.

Le dépôt contient une application Flutter dans lib/ et une API FastAPI dans backend/. Les ressources graphiques sont placées dans assets/images/, et les dossiers Android, iOS et Web permettent les builds Flutter.

2. Objectifs fonctionnels

Compte et session

  • Créer un compte utilisateur.
  • Se connecter avec e-mail et mot de passe.
  • Conserver la session localement pendant 30 jours.
  • Modifier certaines informations de profil.

Courses et recettes

  • Ajouter des aliments manuellement.
  • Analyser une image de liste de courses.
  • Calculer un bilan nutritionnel global.
  • Rechercher et sauvegarder des recettes.

Le produit propose aussi un début de parcours d'abonnement Premium via Stripe Checkout.

3. Périmètre actuel

Fonctionnel

  • Authentification FastAPI avec token JWT.
  • Stockage local du token, de l'utilisateur et de l'expiration de session.
  • Navigation principale en cinq onglets avec conservation de l'état.
  • OCR côté backend avec RapidOCR.
  • Filtrage alimentaire par TheMealDB, alias français et fallback local.
  • Recherche de recettes via TheMealDB.
  • Favoris persistés localement.
  • Création d'une session Stripe Checkout.

Partiel ou à finaliser

  • Suggestions de menus réellement basées sur la liste de courses.
  • Webhook Stripe pour passer automatiquement en Premium.
  • Restrictions Free/Premium.
  • Historique complet des listes.
  • Synchronisation cloud des favoris.

4. Architecture générale

Utilisateur | v Application Flutter |-- UI : auth, accueil, liste, recettes, favoris, profil |-- Providers : session, favoris, liste |-- Services : API Komi, OCR, profil, paiement, recettes | +--> API Komi FastAPI | |-- Authentification JWT | |-- Utilisateurs et profils | |-- OCR liste de courses | |-- Filtrage alimentaire | |-- Abonnement Stripe | +--> PostgreSQL ou SQLite | +--> TheMealDB |-- Recettes |-- Ingrédients de référence

Cette séparation garde l'application mobile centrée sur l'expérience utilisateur, tandis que les traitements sensibles ou lourds restent côté serveur.

5. Frontend Flutter

Flutter Provider SharedPreferences HTTP Dio ImagePicker UrlLauncher

Le point d'entrée lib/main.dart lance KomiApp. L'application initialise trois providers : FavoritesProvider, ShoppingListProvider et UserSessionProvider.

La navigation principale utilise un IndexedStack, ce qui permet de conserver l'état des pages lorsque l'utilisateur change d'onglet.

6. Parcours utilisateur

Inscription et connexion

Le front appelle POST /api/v1/auth/register ou POST /api/v1/auth/login. L'API renvoie un token JWT et un objet utilisateur, ensuite stockés localement.

Accueil

L'accueil affiche l'utilisateur, le bilan des courses, la liste actuelle et des suggestions de recettes. Le score nutritionnel est calculé localement depuis les produits présents.

Liste de courses

La liste accepte une saisie manuelle ou une image. Le backend renvoie les aliments acceptés et les lignes rejetées. Les aliments sont ajoutés à la liste courante.

Recettes et favoris

Les recettes proviennent directement de TheMealDB. Les favoris sont encodés en JSON et stockés localement.

Profil et abonnement

Le profil peut être mis à jour partiellement. Le bouton Premium crée une session Stripe Checkout et ouvre l'URL de paiement dans le navigateur.

7. Backend FastAPI

Le backend expose une API versionnée sous /api/v1 et une route /health. Il peut fonctionner avec PostgreSQL via Docker ou SQLite en local simple.

DomaineEndpoints principaux
AuthPOST /auth/register, POST /auth/login, POST /auth/forgot-password, POST /auth/reset-password
UtilisateurGET /users/me, PATCH /users/me, POST /users/change-password
CoursesPOST /shopping-lists/analyze-image, POST /shopping-lists/validate-items
AbonnementGET /subscription/me, PUT /subscription/me, POST /subscription/checkout/premium
FacturesGET /invoices/me, POST /invoices/me/demo

Les routes privées utilisent get_current_user, qui décode le bearer token et récupère l'utilisateur correspondant en base.

8. Modèle de données

User

Identité, e-mail unique, profil, avatar, hash du mot de passe et date de création.

Subscription

Plan, statut, date de renouvellement et lien unique vers l'utilisateur.

Invoice

Référence, date, libellé, montant en centimes, statut et utilisateur lié.

PasswordResetToken

Hash du token, expiration, date de création et date d'utilisation.

SQLAlchemy permet de garder le même code métier avec PostgreSQL en Docker et SQLite en local.

9. OCR et filtrage alimentaire

Image envoyée -> contrôle du fichier -> préparation de variantes d'image -> RapidOCR sur chaque variante -> choix du meilleur résultat -> nettoyage des lignes -> filtrage alimentaire -> réponse : items + rejectedItems

Le backend prépare plusieurs versions de l'image : originale, contrastée, renforcée, binaire et binaire inversée. Il sélectionne ensuite la meilleure analyse selon le nombre d'aliments reconnus, la confiance OCR et la quantité de texte détectée.

Le filtrage alimentaire utilise TheMealDB, une liste locale de termes français et des alias français vers anglais. Les textes non alimentaires sont rejetés avant l'ajout à la liste.

10. Sécurité

  • Mots de passe hashés avec bcrypt.
  • Politique de mot de passe : longueur, majuscule, minuscule, chiffre et caractère spécial.
  • Refus de mots de passe communs ou contenant le nom/e-mail.
  • JWT signés pour les sessions API.
  • Routes privées protégées par bearer token.
  • Tokens de réinitialisation stockés sous forme hashée.
Avant production : remplacer la clé JWT par défaut, restreindre CORS, désactiver le lien de reset exposé en réponse, ajouter le webhook Stripe signé et limiter les tailles d'images.

11. Paiement et abonnement

Le plan Premium est prévu à 6 euros par mois. Le backend crée une session Stripe Checkout en mode abonnement avec STRIPE_PREMIUM_PRICE_ID.

Clic Premium -> POST /subscription/checkout/premium -> création Stripe Checkout Session -> retour URL Stripe -> ouverture navigateur externe

Le webhook checkout.session.completed reste à implémenter pour mettre à jour automatiquement le compte utilisateur après paiement.

12. Tests et qualité

Les tests Flutter existants vérifient la splash page et le rendu des écrans d'authentification sur plusieurs tailles mobiles, afin de détecter les problèmes d'overflow.

Tests recommandés à ajouter : score nutritionnel, session locale expirée, inscription/connexion backend, validation alimentaire, et parcours login + ajout manuel.

13. Choix de conception

  • Flutter : adapté à une expérience mobile avec possibilité de démonstration web.
  • Provider : gestion d'état simple, suffisante pour le MVP.
  • Backend dans le même dépôt : facilite le lancement par l'équipe.
  • OCR côté serveur : allège l'application et simplifie les améliorations futures.
  • TheMealDB : accélère la création d'un parcours recettes complet.

14. Limites et évolutions

Limites connues

  • Liste de courses non persistée côté backend.
  • Données nutritionnelles génériques pour les produits inconnus.
  • Favoris stockés seulement localement.
  • Suggestions non encore calculées à partir de la liste réelle.
  • Premium non activé automatiquement après paiement.

Évolutions recommandées

  1. Persister les listes de courses côté backend.
  2. Ajouter le webhook Stripe.
  3. Synchroniser favoris et listes avec le compte utilisateur.
  4. Connecter une base produits plus complète, par exemple Open Food Facts.
  5. Ajouter des tests backend et des tests d'intégration.
